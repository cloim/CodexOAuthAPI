# Windows 로그인 후 서버 자동 시작 설계

## 목적

`<WINDOWS_USER>` 사용자가 Windows에 로그인하면 CodexOAuthAPI 서버를 자동으로 시작한다. 서버는 로컬 인터페이스 `127.0.0.1`의 `47307` 포트에서 실행한다.

## 범위

- Windows 작업 스케줄러에 사용자 로그인 트리거 작업을 등록한다.
- 프로젝트 작업 폴더와 가상환경 `pythonw.exe` 실행 파일을 절대 경로로 고정한다.
- 동일 서버의 중복 실행을 막고, 비정상 종료 시 제한된 횟수만큼 자동 재시작한다.
- 작업 등록 직후 수동 실행으로 기동 상태와 HTTP 상태 확인을 검증한다.

시작프로그램 폴더, 레지스트리 `Run`, Windows 서비스 등록은 범위에 포함하지 않는다.

## 실행 계약

- 실행 계정: `<WINDOWS_HOST>\<WINDOWS_USER>`
- 예약 작업 이름: `CodexOAuthAPI-47307`
- 트리거: `<WINDOWS_USER>` 사용자 로그인
- 작업 폴더: `<PROJECT_ROOT>`
- 실행 파일: `<PROJECT_ROOT>\.venv\Scripts\pythonw.exe`
- 실행 인수: `-m codex_oauth_api.main serve --host 127.0.0.1 --port 47307`
- 표시 방식: 로그인 화면에 콘솔 창을 띄우지 않는 백그라운드 실행
- 중복 실행 정책: `IgnoreNew`로 기존 인스턴스 유지
- 실패 정책: 1분 간격으로 최대 3회 재시작
- 실행 시간 제한: 없음

프로젝트의 `.env`와 `.codex-oauth-api-state`는 작업 폴더 기준으로 그대로 사용한다. `pythonw` 실행 시 없는 stdout/stderr는 `codex_oauth_api.main`이 `.codex-oauth-api-state` 아래 로그 파일로 연결한다. 경로나 인수는 다른 후보를 탐색하는 fallback 없이 위 계약과 정확히 일치시킨다.

## 등록 및 검증 순서

1. 같은 이름의 예약 작업과 `47307` 포트 점유 여부를 확인한다.
2. 사용자 로그인 트리거와 실행 계약을 포함한 예약 작업을 등록한다.
3. 예약 작업을 수동으로 한 번 시작한다.
4. 작업 상태, 실행 프로세스, `127.0.0.1:47307` 수신 상태를 확인한다.
5. `GET /health`가 성공 응답을 반환하는지 확인한다.

## 오류 처리

- 예약 작업 등록 권한이 부족하면 관리자 승인으로 등록을 다시 수행한다.
- 실행 파일, 프로젝트 폴더, 포트 또는 HTTP 상태 확인이 계약과 다르면 완료로 처리하지 않고 실제 오류를 보고한다.
- 기존에 동일 목적의 작업이나 프로세스가 발견되면 충돌 여부를 먼저 확인한 뒤 하나의 인스턴스만 유지한다.

## 완료 조건

- 예약 작업의 이름, 트리거, 계정, 작업 폴더, 실행 파일, 인수와 실행 정책이 실행 계약과 일치한다.
- 예약 작업을 수동 시작했을 때 `47307` 포트에서 서버가 수신한다.
- `http://127.0.0.1:47307/health`가 성공 상태를 반환한다.
- 재등록 없이 다음 `<WINDOWS_USER>` 로그인부터 같은 서버가 자동 시작될 수 있다.
